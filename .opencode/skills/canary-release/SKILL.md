# AALC Canary Release

## Release stance

Canary release work writes to git, tags, CI, and GitHub Releases. Treat every external
write as intentional release work: verify first, then ask for clear confirmation before
pushing tags or editing remote releases.

Do not read, print, paste, or commit tokens. Prefer an already-authenticated `gh` CLI.
If API access is needed, pass credentials through environment variables or an approved
credential helper without echoing secret values.

## Preflight

1. Confirm the target version, for example `X.Y.Z-canary.N`.
2. Run `git status --short` and preserve unrelated user changes.
3. Confirm the release scope and changelog entries with the user if they are unclear.
4. Verify `bootstrap_version` only changes for protocol incompatibility.
5. Keep generated update protocol files generated by the build scripts; do not hand-edit
   sidecar manifests.

## Files commonly touched

- `assets/config/version.txt` for the canary version.
- `CHANGELOG.md` for user-visible release notes.
- Build or update scripts only when the release itself requires a code fix.

Do not update user `config.yaml`.

## Local verification before remote writes

Choose the narrowest checks that match the touched files:

```powershell
uv run python -m py_compile scripts\build.py
uv run python .\scripts\build.py --version dev
uv run python .\scripts\check_i18n.py --update
```

If `scripts/build.py` was touched, its terminal output must remain pure ASCII because
Windows CI may run with cp1252.

## Commit, tag, and push gate

Before any push or tag:

1. Show the planned version, commit message, tag, and remotes.
2. Show `git diff --stat` and the relevant changelog/version diff.
3. Ask the user for explicit confirmation.

Only after confirmation:

```powershell
git add assets/config/version.txt CHANGELOG.md
git commit -m "发布 canary X.Y.Z-canary.N"
git tag -a "vX.Y.Z-canary.N" -m "vX.Y.Z-canary.N"
git push origin HEAD
git push origin "vX.Y.Z-canary.N"
```

Adjust the staged files only to match the actual release diff.

## CI and GitHub Release

After the tag triggers CI:

1. Wait for the release workflow to finish.
2. Verify the release contains exactly one `AALC_<version>.7z` and one
   `AALC.update_manifest.json`.
3. For official stable releases, also verify `AALC_<version>.7z.sha256`; for canary,
   follow the current project release policy.
4. Confirm the release is marked prerelease so stable-channel users do not receive it.

Prefer `gh release edit` with a notes file:

```powershell
gh release edit "vX.Y.Z-canary.N" `
  --title "vX.Y.Z-canary.N - 金丝雀预览版" `
  --notes-file release-notes.md `
  --prerelease
```

If GitHub API fallback is required, generate JSON with a structured encoder such as
Python `json.dumps`; do not hand-build JSON around Markdown with ad hoc escaping.

## Release notes

Release notes should come from `CHANGELOG.md` and the GitHub Release body. Keep them
user-visible and concise:

- New behavior or fixes users can observe.
- Known risks for canary testers.
- Upgrade or rollback guidance when relevant.

## Common failure modes

| Symptom | Likely cause | Response |
|---|---|---|
| Windows CI `UnicodeEncodeError` | Non-ASCII console output in build scripts | Replace output text with ASCII and rerun the relevant check. |
| Release notes missing or stale | CI-created release was not edited after upload | Use `gh release edit` and verify the rendered body. |
| Stable users see canary | Release is not marked prerelease or channel filtering changed | Stop and fix release metadata/update-channel logic before announcing. |
| Update manifest mismatch | Hand-edited or stale generated protocol file | Regenerate through scripts; do not patch sidecar manifest manually. |

## Final report

Report:

- Version and tag.
- Commit hash.
- CI workflow result.
- Release URL and artifact list.
- Verification commands run.
- Any manual follow-up still required.
